# 設計メモ：サークル作品ホスティングサービス

---

## 1. 何を解決したい？

- **困っていること（いま何が問題？）：**
  - サークルメンバーが制作したWebアプリ（バックエンド常時稼働が必要なもの）を公開・運用できる場所がない
  - 無料ホスティング（Vercel、Cloudflare Pages等）は静的サイト向けで、常時稼働Webサーバーには対応しにくい

- **既存サービスを使った解決策とその問題点：**

  | 既存サービス | できること | 問題点 |
  |-------------|-----------|--------|
  | Vercel / Cloudflare Worker | 静的サイト・SSR | 常時稼働Webサーバーは不可 |
  | Render（無料枠） | Webサービス | 無料枠はスリープする |
  | Railway / Fly.io | コンテナホスティング | 無料枠が小さい  |

- **だれが困る？：**
  - サークルメンバー（10名以上）：作品を公開・デモできない
  - 作品の閲覧者：ポートフォリオや文化祭で作品を見られない

- **うまくいったと言える条件（できれば数字）：**
  - 同時に10作品以上を常時稼働でホスティングできている
  - メンバー1人あたり月額0円（電気代＋ドメイン代 年〜$10 のみサークル費）

---

## 2. 今回やらないこと

- やらないこと：
  - 静的サイトのホスティング（無料サービスで十分）
  - マルチサーバー構成・冗長化（1台で始める）
  - 独自ドメインの個別対応（サブドメイン割当のみ）
  - Git pushによる自動デプロイ（Phase 4以降）
  - 管理Web UIの実装（Phase 3以降。Phase 1〜2は管理者が手動セットアップ）

---

## 3. アイデア

- **こうする：**
  - 自宅サーバーに Proxmox VE を入れ、作品ごとに独立した **LXC コンテナ**（Debian 12）を作成する
  - 言語ごとに1度整備したテンプレートを `pct clone` で複製するだけでデプロイできる
  - アプリは systemd サービスとして直接実行
  - 管理用 LXC に Caddy + Cloudflare Tunnel を配置し、ポート開放ゼロで外部公開する
  - 管理 Web UI は Cloudflare Workers（Hono）+ Pages（React）+ D1 で実装し、Proxmox API を操作する

- **なぜそうする？（理由は3個）：**
  1. Proxmox 公式の推奨範囲内：LXC 内で Docker / Podman を動かすことは公式が非推奨（開発者 Fabian がフォーラムで明言）。LXC 直接運用はその制約を受けず、Proxmox アップデートで突然壊れるリスクがゼロ
  2. クローンで手間を最小化：毎回ゼロから環境構築するのではなく、言語別テンプレートを `pct clone` 1コマンドで複製する。Phase 3以降は管理UIのボタン1つで全自動になる
  3. 管理UIも追加コストゼロ：Cloudflare Workers / Pages / D1 はすべて無料枠内で運用できる。サーバーを別途用意する必要がない

---

## 4. 他の案

- 案A：LXC + Docker Engine
  - 良いところ：Dockerfile がそのまま使える。コミュニティの情報が多い
  - 悪いところ：Engine 常駐で〜300MB 消費。Proxmox 公式非推奨（nesting 設定が必要でアップデートで壊れるリスクあり）

- 案B：LXC + Podman
  - 良いところ：デーモンレスでメモリ効率が良い。Dockerfile が使える
  - 悪いところ：同じく Proxmox 公式非推奨。rootless 動作に複雑な UID/GID マッピングが必要。再起動後のキャッシュ削除バグの報告あり

- 案C：Kubernetes
  - 良いところ：自動スケーリング・ヘルスチェック標準装備
  - 悪いところ：学習コスト極大。1ヶ月では完走不可能

- 今回選ばない理由（短く）：
  - 案A・B：Proxmox アップデートで動作が壊れるリスクを許容できない
  - 案C：リソースと学習コストが見合わない。期限内に完走できない

## 5. 導入によって生じるサークル内の変化

- フローの変化：
  - Before：作品を作っても公開手段がない → ローカルでデモする
  - After：管理UIのフォームで申請 → 管理者がボタン1つで承認・デプロイ → `{作品名}.example.dev` で即日公開

- サークル全体での変化（効率面/費用面）：
  - Before：運用費用が個人で月額かかってしまう
  - After：電気代（月〜数百円）＋ドメイン代のみ。管理UIのホスティングも無料

- 利用者個人の変化（心理的側面）：
  - Before：「作ったけど見せられない」→ モチベーション低下
  - After：「URLを共有するだけで見てもらえる」「就活ポートフォリオに載せられる」→ 完成まで持っていく動機が明確になる

## 6. 作業の進め方

> インフラ（Phase 1〜2）を先に動かしてから、管理UI（Phase 3）を作る。

### 6.1 全体の見積もり

| フェーズ | 作業内容 | 担当 | 見積もり | 期間 |
|---------|---------|------|---------|------|
| Phase 1 | Proxmox 構築・テンプレート作成・Cloudflare Tunnel | 柳井 | 15〜25h | 2〜3週 |
| Phase 2 | デプロイスクリプト整備 | 柳井 | 8〜12h | 1〜2週 |
| Phase 3 | 管理Web UI実装（Remix） | 柳井 | 10〜20h | 2〜4週 |
| Phase 4 | Git push 自動デプロイ| 柳井 | 15〜20h | 将来 |

- **全体：約50〜75時間（Phase 4含む）**
- 使える日：（要記入）

### 6.2 最初にやること（1〜3日で終わるサイズ）

- **まず作る最小版（MVP）：**
  1. Proxmox VE をインストール
  2. No-Subscription リポジトリ追加・アップデート
  3. Debian 12 LXC を1つ作成
  4. LXC 内に Node.js 22 をインストール → サンプルアプリを systemd で起動
  5. `pct clone` でテンプレート複製が動くことを確認
  6. Cloudflare Tunnel 経由でブラウザからアクセス確認

- **それで確かめたいこと：**
  - Debian 12 LXC で各言語ランタイムが正常に動くか
  - systemd がメモリ・再起動を正しく管理できるか
  - `pct clone` でテンプレートが正常に複製できるか

- **できなかったらどうする（簡単な逃げ道）：**
  - 特定言語が LXC で動かない → その言語のみ LXC + Docker Engine（nesting 有効化）で対応
  - Proxmox VE 自体が難しい → ベアメタル Ubuntu Server + systemd 単体構成にフォールバック
  - Phase 3 の管理UI が複雑になりすぎる → HTML フォーム + シェルスクリプト Webhook で代替
